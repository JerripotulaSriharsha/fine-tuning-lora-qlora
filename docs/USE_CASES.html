3. Primary Use Cases

UC-01: Single Customer Credit Risk Assessment

Actor Credit Analyst (Sarah)
Interface Web UI
Frequency 20-50 times/day
Priority High

Goal

Assess creditworthiness of a loan applicant

Preconditions

  • User has loan application form
  • Web UI is accessible and models are loaded
  • User has necessary customer financial data

Main Flow

1. User opens Streamlit web application
2. User enters customer information:
  • Age: 32
  • Occupation: Journalist
  • Annual Income: $33,470.43
  • Outstanding Debt: $1,318.49
  • Credit Utilization: 26.8%
  • Payment Behavior: High_spent_Small_value_payments
3. User selects "QLoRA" model for assessment
4. User clicks "Generate Credit Risk Assessment" button
5. System formats input and sends to QLoRA model
6. System streams model response in real-time
7. User views prediction: "Good" with detailed reasoning
8. User reviews reasoning section for context
9. User records assessment in loan management system

Alternative Flow: Unexpected Classification

7a. User is surprised by "Bad" classification
7b. User reviews reasoning section
7c. User identifies high debt-to-income ratio as key factor
7d. User requests additional documentation from applicant
Expected Outcome:
Response time: < 3 seconds | Clear classification: Good/Bad/Standard | Transparent reasoning provided

UC-02: Multi-Model Comparison

Actor Credit Analyst (Sarah)
Interface Web UI
Frequency 5-10 times/day
Priority High

Goal

Compare predictions across multiple models for high-value loan (>$100K)

Main Flow

1. User enters customer financial information
2. User selects all three checkboxes: QLoRA, LoRA, and Base
3. User clicks "Generate Credit Risk Assessment"
4. System runs all three models in sequence
5. System displays three color-coded containers:
  • Blue container shows QLoRA prediction: "Standard"
  • Orange container shows LoRA prediction: "Good"
  • Purple container shows Base prediction: "Bad"
6. User analyzes disagreement between models
7. User reviews reasoning from each model
8. User identifies LoRA as most accurate based on 60% track record
9. User makes final decision based on LoRA prediction
10. User documents model consensus/disagreement in notes
Expected Outcome:
Total processing time: < 10 seconds | Clear comparison view | Informed decision-making

UC-03: Batch Credit Assessment via API

Actor Backend Developer (Alex)
Interface REST API
Frequency Daily batch job
Priority High

Goal

Process 500 credit applications overnight

Main Flow

1. Cron job triggers batch processing script at 2:00 AM
2. Script queries database for pending applications
3. For each application:
  • Extract customer financial data
  • Format API request payload
  • Call POST /inference/lora endpoint
  • Parse response JSON
  • Extract classification and reasoning
  • Update database with prediction
  • Log transaction with timestamp
4. Script completes processing all 500 records
5. Script generates summary report: 
Total processed: 500
Good: 150 (30%)
Standard: 250 (50%)
Bad: 100 (20%)
Average latency: 2.3 seconds
Failed requests: 0
6. Script emails report to operations team

Alternative Flow: API Timeout

3d. Request exceeds 10-second timeout
3e. Script logs error with customer ID
3f. Script retries request (max 3 attempts)
3g. If still fails, mark for manual review
3h. Continue with next application
Expected Outcome:
99.9% success rate | Average latency < 3 seconds per request | Zero data loss

4. Secondary Use Cases

UC-05: Model Performance Monitoring

Actor Data Scientist (Marcus)
Interface Jupyter Notebook
Frequency Weekly review
Priority Medium

Goal

Monitor model accuracy and detect drift

Main Flow

1. Data scientist opens evaluation.ipynb notebook
2. Loads 10-sample test set (evaluation_examples.json)
3. Runs all three models on test set
4. Calculates accuracy metrics:
Model Accuracy Correct
Base Model 20% 2/10
QLoRA Model 50% 5/10
LoRA Model 60% 6/10
5. Compares to previous week's results
6. Identifies LoRA accuracy dropped from 70% to 60%
7. Investigates potential causes (distribution shift, model corruption, pipeline changes)
8. Documents findings in model registry
9. Schedules retraining if drift threshold exceeded
Expected Outcome: Early detection of model degradation | Data-driven retraining decisions

UC-06: Model Explainability for Loan Denial Appeal

Actor Credit Analyst + Applicant
Interface Web UI
Frequency 5-10 times/week
Priority Medium

Goal

Explain denial decision to applicant

Main Flow

1. Applicant appeals loan denial via customer service
2. Customer service transfers applicant to credit analyst
3. Analyst retrieves original application data
4. Analyst re-runs assessment in web UI with applicant on phone
5. Analyst reads model reasoning: 
<reasoning>
The customer has an outstanding debt of $4,072 against an annual
income of $18,667, resulting in a debt-to-income ratio of 21.8%.
Combined with a credit utilization of 26.7% and low-spent medium-value
payment behavior, this indicates financial stress. The age of 37 with
mechanic occupation suggests limited income growth potential.
</reasoning>
<answer>Bad</answer>
6. Analyst explains key factors in plain language:
  • "Your debt is about 22% of your annual income, which is concerning"
  • "You're using 27% of your available credit"
  • "Your payment history shows some late payments"
7. Analyst provides actionable advice:
  • Pay down $2,000 of your debt
  • Reduce credit utilization below 20%
  • Make on-time payments for next 6 months
8. Analyst schedules follow-up reapplication in 6 months
Expected Outcome: Transparent explanation | Applicant understanding | Clear path to reapplication | Reduced complaints

6. Research and Development Use Cases

UC-10: Transfer Learning - New Credit Product

Actor Research Data Scientist
Interface Jupyter Notebook
Frequency Quarterly
Priority Low

Goal

Adapt model for auto loan risk assessment

Main Flow

1. Data scientist receives 5,000 auto loan applications
2. Labels data: Good, Bad, Standard
3. Opens train_qlora.ipynb notebook
4. Modifies dataset path to auto loan data
5. Adds auto-specific features:
  • Vehicle value
  • Loan-to-value ratio
  • Vehicle age
6. Keeps existing LoRA configuration (rank=32)
7. Trains for 50 steps on auto loan data
8. Evaluates on 100-sample test set
9. Compares to baseline:
  • Base model: 25%
  • Fine-tuned model: 58%
10. Deploys auto loan model to separate API endpoint
Expected Outcome: Rapid domain adaptation | Minimal training data (5K vs 30K) | Separate model per product

UC-11: Bias Detection and Mitigation

Actor Fairness ML Engineer
Interface Jupyter Notebook
Frequency Quarterly audit
Priority High

Goal

Detect and mitigate demographic bias

Main Flow

1. Engineer loads evaluation dataset with demographic labels
2. Stratifies test set by age groups and occupation
3. Runs model predictions on each stratum
4. Calculates per-group accuracy:
Age Group Accuracy
<25 45%
25-40 62%
40-60 58%
60+ 51%
5. Identifies underperformance for young applicants
6. Investigates training data: Age <25 only 12% of data
7. Implements mitigation: Upsample young examples, add penalty
8. Retrains model with balanced data
9. Re-evaluates: Age <25 accuracy improves to 57%
10. Documents bias audit in model card
Expected Outcome: Fair performance across demographics | Regulatory compliance | Reduced discrimination risk

7. Edge Cases and Error Scenarios

UC-12: Handling Invalid Input Data

Actor Backend Developer
Interface REST API
Frequency 1-2% of requests
Priority High

Goal

Gracefully handle malformed API requests

Main Flow

1. API receives request with invalid age: -5
2. Pydantic validation layer catches error
3. API returns 422 Unprocessable Entity: 
{
  "detail": [{
    "loc": ["body", "age"],
    "msg": "ensure this value is greater than 18",
    "type": "value_error.number.not_gt"
  }]
}
4. Client application parses error response
5. Client displays user-friendly message: "Invalid age entered. Please enter age between 18-100."
6. User corrects input and resubmits
Expected Outcome: Clear error messages | No server crashes | User can self-correct
